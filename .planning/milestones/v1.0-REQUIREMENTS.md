# Requirements: fomod-installer Linux/Proton Compatibility Fork

**Defined:** 2026-04-09
**Core Value:** FOMOD mods install correctly on Linux/Vortex with no silent partial installs and no Vortex-side workarounds required.

## v1 Requirements

Requirements for initial release. Each maps to roadmap phases.

### Path Normalization

- [x] **PATH-01**: All XML parser classes call `TextUtil.NormalizePath(value, false, true)` on every `source` and `destination` attribute extracted from FOMOD XML, covering all versioned parsers (Parser10.cs, Parser20.cs, and any others in `src/InstallScripting/XmlScript/Parsers/`)
- [x] **PATH-02**: `Mod.cs` emits the matched archive-case path (real case from archive) in copy instructions, not the XML-verbatim path — eliminating case-sensitivity file-not-found failures on Linux ext4

### Script Guard

- [x] **GUARD-01**: CSharpScript type registration is gated behind `RuntimeInformation.IsOSPlatform(OSPlatform.Windows)` in `ModFormatManager.cs` (or `ScriptTypeRegistry.cs`) so the IPC process does not attempt to load Windows-only assemblies on Linux

### IPC Build Pipeline

- [x] **IPC-01**: GitHub Actions `build-packages.yml` adds a Linux runner (`ubuntu-22.04`) to the IPC build matrix that publishes a self-contained single-file ELF binary via `dotnet publish -r linux-x64 --self-contained true -p:PublishSingleFile=true -p:EnableCompressionInSingleFile=true`, sets execute permissions (`chmod +x`) before packaging, and includes the binary in the npm tarball under a platform-specific path (e.g. `dist/linux-x64/ModInstallerIPC`)
- [x] **IPC-02**: `RegularProcessLauncher.ts` resolves the correct IPC binary path based on `process.platform` and `process.arch`, and `BaseIPCConnection.findExecutable()` no longer hardcodes `.exe` unconditionally — ELF binary is resolved on Linux without Mono fallback
- [x] **IPC-03**: `cleanup-processes.ts` is cross-platform — replaces `tasklist`/`taskkill` with platform-aware process detection (`pgrep`/`kill` on Linux) so it does not throw `ENOENT` on Linux and correctly cleans up orphaned IPC processes

### UX & Documentation

- [x] **UX-01**: `Instruction.cs` `UnsupportedFunctionalityWarning()` carries a `reason` field (e.g. `"CSharpScript not supported on Linux"`) and a `platform` field so callers can construct actionable OS-specific user messages
- [x] **UX-02**: `README.md` documents Linux limitations: C# script Windows-only restriction, native package as recommended Linux path, IPC package Linux support from which version, and Vortex workarounds that can be removed after this fork's fixes land
- [x] **UX-03**: Unit tests cover `FileSystem.IsSafeFilePath()` with `../` and `..\` traversal sequences on Linux, confirming path traversal protection works on the Linux `Path.GetInvalidPathChars()` set

## v2 Requirements

Deferred to future release. Tracked but not in current roadmap.

### SteamOS Integration

- **STEAM-01**: Errors propagate with sufficient context for callers to differentiate Steam Deck vs generic Linux failures and surface polkit/privilege warnings (requires Vortex-side coordination)

### Extended Platform Coverage

- **PLAT-01**: `linux-arm64` self-contained IPC binary in CI matrix (Steam Deck is x86_64; ARM64 is future)

## Out of Scope

| Feature | Reason |
|---------|--------|
| C# FOMOD script execution on Linux (Mono or Roslyn) | Scripts reference `System.Windows.Forms`, `System.Drawing.Common`, Windows registry APIs — cannot be shimmed. Silent wrong results are worse than an explicit unsupported warning |
| Case-insensitive filesystem shim (FUSE/overlay) | System-level dependency outside fomod-installer control; not available on Steam Deck immutable OS |
| Breaking API changes | Changes must be PR-splittable and merge-compatible with upstream `Nexus-Mods/fomod-installer` |
| Named pipe transport on Linux | Named pipes are unavailable on Linux; TCP is the correct primary strategy |
